\section{Overview}
This assignment pretrains a GPT-style language model on the Chinese Wikipedia subset of CLUECorpus2020. The pipeline has four major parts:
\begin{enumerate}[leftmargin=1.5em]
  \item extract and preprocess the raw JSON-lines corpus from the provided zip file;
  \item train a new BPE tokenizer from scratch on the extracted corpus;
  \item complete the pretraining script so that it supports streamed loading, HuggingFace tokenizers, checkpointing, evaluation, and resuming;
  \item pretrain the model, save the final weights, and report the final results.
\end{enumerate}

\section{Requirement 1: Data Extraction and Preprocessing}
I implemented \texttt{extract\_wikizh.py} to read the original \texttt{wiki\_zh\_2019.zip} archive directly. Each raw file inside the archive contains one JSON object per line. For every JSON record, the script concatenates the \texttt{title} and \texttt{text} fields, skips malformed lines safely, and appends the special separator token \texttt{<|endoftext|>} after each article.

Instead of merging the entire corpus into one huge text file, I preserved the original shard structure and wrote one \texttt{.txt} file per input shard. This is more memory-efficient during downstream tokenization and training because each shard can be processed independently.

\subsection{Tokenizer Comparison}
To compare the new tokenizer with the original GPT-2 tokenizer, I used the sample sentence ``太阳照常升起。'' The trained tokenizer segments this Chinese sentence much more compactly than the original GPT-2 tokenizer.

\begin{lstlisting}
Sample text: 太阳照常升起。
Trained tokenizer token count: 5
Trained tokenizer IDs        : [24932, 1406, 647, 30317, 217]
GPT-2 tokenizer token count  : 16
GPT-2 tokenizer IDs          : [13783, 103, 165, 246, 111, 163, 227, 100,
                                30585, 116, 39355, 229, 164, 113, 115, 16764]
\end{lstlisting}

For byte-level tokenizers, individual token strings are often displayed as UTF-8 byte fragments rather than human-readable Chinese characters. The important result is that the tokenizer trained on Chinese data requires far fewer tokens for the same sentence.

\section{Requirement 3: Completing the Pretraining Script}
I completed \texttt{run\_pretrain.py} and updated \texttt{utils.py}. The main additions are:
\begin{itemize}[leftmargin=1.5em]
  \item support for loading HuggingFace tokenizer JSON files via \texttt{tokenizers.Tokenizer};
  \item memory-efficient streaming dataloaders for both raw text shards and pre-tokenized \texttt{.npy} shards;
  \item a \texttt{global\_step} counter for periodic evaluation and checkpoint saving;
  \item periodic logging to \texttt{train.log} and \texttt{metrics.jsonl};
  \item learning-rate warmup followed by cosine decay;
  \item mixed-precision training on CUDA;
  \item interrupt-safe checkpoints and resume-from-checkpoint support.
\end{itemize}

The assignment required keeping the original GPT-2-small-style architecture and only changing the vocabulary size to match the tokenizer. The final model configuration is shown below.

\subsection{Final Results}
During online training, evaluation was intentionally lightweight (\texttt{eval\_iter=1}) so that logging overhead stayed low. This makes the validation curve somewhat noisy. The best online validation loss recorded in the training log was:
\[
\texttt{val\_loss} = 5.1244
\]
at global step 79,000 after seeing 242,688,000 training tokens.

No additional training was performed after the single 1-epoch run. To obtain a more reliable final estimate without retraining, I reloaded the final checkpoint from that same run and evaluated it again on more held-out validation mini-batches after training ended. The post-training results are:

\section{Conclusion}
This assignment produced a complete end-to-end Chinese GPT pretraining pipeline: raw corpus extraction, Chinese-specific tokenizer training, streamed pretraining with checkpointing and resume support, and final model export. The trained tokenizer clearly improves Chinese tokenization over the original GPT-2 tokenizer, and the final model reaches a stable post-training validation loss below 5.0 on held-out evaluation batches.
